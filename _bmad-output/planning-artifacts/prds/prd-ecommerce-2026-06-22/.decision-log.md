## Decisions Captured

### D1: MVP Scope Focused on Stability, Not Scale
**Decision**: Keep MVP feature set intentionally narrow; prioritize stability (0 critical bugs, 99%+ order success) over feature expansion.

**Rationale**: Solo developer + 4-6 week timeline means deep, well-architected core beats broad but fragile feature set. Market validation comes from reliable experience, not feature breadth.

**Implications**: 
- Out-of-scope features deferred to v1.1 (reviews, recommendations, multi-currency, marketplace)
- Development prioritizes testability, monitoring, and error handling
- Performance targets are realistic (100-500 concurrent users, not viral scale)

**Status**: ✅ Captured in PRD (Executive Summary, Success Metrics, Out of Scope)

---

### D2: Required Registration (No Guest Checkout in MVP)
**Decision**: Users must create an account and verify email before checkout.

**Rationale**: Simplifies order management, enables future personalization, reduces friction from multiple checkout flows in MVP.

**Implications**:
- Registration form is part of critical path (must be fast and frictionless)
- Email verification required (OTP or link)
- Account recovery (password reset) needed

**Status**: ✅ Captured in PRD (User Auth, Checkout Flow, Assumptions)

**Open for revisit**: Guest checkout could move to v1.1 if user feedback shows high bounce rate.

---

### D3: Mock Payments (Stripe Test Mode) in MVP
**Decision**: Use Stripe test mode or mock payment endpoint; no real money processing until production handoff.

**Rationale**: Reduces PCI compliance burden in development; safe testing; clear separation of concerns (business logic first, payment integration after validation).

**Implications**:
- Payment flow tested with test cards only
- Refund logic deferred to v1.1
- Full PCI compliance audit scheduled for production launch

**Status**: ✅ Captured in PRD (Payment & Billing, Assumptions)

---

### D4: Single Country, Flat Shipping, Flat Tax
**Decision**: MVP supports single country (no multi-currency); hardcoded flat shipping fee ($5); hardcoded flat tax rate (~10%).

**Rationale**: Eliminates internationalization, carrier integration, and tax calculation complexity; achievable in 4-6 weeks; can scale to multi-country in v1.1.

**Implications**:
- Checkout simplified (no currency selection, no tax calculation)
- Shipping integration deferred
- Geo-targeting and localization deferred

**Status**: ✅ Captured in PRD (Assumptions, Out of Scope)

---

### D5: Stock Deduction on Order Confirmation (Not During Checkout)
**Decision**: Inventory is decremented only after successful payment confirmation, not when item is added to cart.

**Rationale**: Prevents race conditions where stock appears taken but payment fails; allows other customers to reserve during another's checkout.

**Implications**:
- "Add to cart" does not hold inventory
- "Out of stock during checkout" edge case handled with retry/removal logic
- Overselling prevented via validation check at payment time

**Status**: ✅ Captured in PRD (Inventory Management, Edge Cases)

---

### D6: Basic Event Tracking (No Advanced Analytics in MVP)
**Decision**: Log only core funnel events (product_view, add_to_cart, checkout_started, order_completed, payment_failed); no cohort analysis, funnels, or retention metrics in MVP.

**Rationale**: Reduces complexity; provides signal for product-market fit without engineering overhead; advanced analytics (Mixpanel, Amplitude) in v1.1.

**Implications**:
- Simple server-side logging (backend logs or lightweight GA4)
- No client-side tracking complexity
- BI/dashboards deferred

**Status**: ✅ Captured in PRD (Analytics & Monitoring, Assumptions)

---

### D6: WCAG 2.1 Level A (Basic Accessibility), Not Full Audit
**Decision**: Implement WCAG 2.1 Level A compliance (keyboard nav, semantic HTML, color contrast, form errors); defer full audit to v1.1.

**Rationale**: Ensures baseline accessibility without blocking MVP launch; accessibility improvements compound over time; Level A covers most common issues.

**Implications**:
- Semantic HTML required (no <div> soup)
- Color contrast 4.5:1 for text
- Alt text for images
- Keyboard navigation tested manually
- Full accessibility audit (ARIA, screen reader testing) scheduled for post-launch

**Status**: ✅ Captured in PRD (Accessibility)

---

### D7: Admin Role Only (No Multiple Roles in MVP)
**Decision**: Single admin role in MVP; no fine-grained permissions (product manager, order manager, etc.).

**Rationale**: Simplifies authentication and authorization logic; sufficient for solo/small business MVP; role-based access control in v1.1.

**Implications**:
- All authenticated admins have same permissions
- No audit trail for individual user actions (logged by timestamp, not user)
- Multi-user admin environments possible only in v1.1

**Status**: ✅ Captured in PRD (Admin Features, Assumptions)

---

### D8: Journey-Led PRD Structure (Not Feature-Led)
**Decision**: Organize PRD around user journeys (Shopper, Admin) rather than pure feature lists.

**Rationale**: 
- Shopper journey is consumer-facing and emotional; features make sense in context of "what Maya does"
- Admin journey clarifies business workflows
- Edge cases surface naturally from journey narration
- Easier for UX/architecture teams to reason about flows

**Implications**:
- Features grouped under journeys, not alphabetically
- Each journey includes success criteria and edge cases
- Out-of-scope deferred features still clear

**Status**: ✅ Captured in PRD (structure)

---

### D9: Email Vendor Selection — Nodemailer + Gmail SMTP
**Decision**: Use Nodemailer with Gmail SMTP for transactional emails (order confirmation, status updates, password reset).

**Rationale**: Simplest, fastest implementation for MVP. No vendor lock-in, no monthly SaaS fees. Sufficient for <100k emails/month. Upgrade to SendGrid or AWS SES post-launch if volume grows.

**Implications**:
- Setup: Gmail account + app-specific password in .env
- Configuration: Email templates in Nodemailer (inline HTML)
- Tracking: Basic delivery (no open/click tracking in MVP)
- Upgrade path: Switch to SendGrid/AWS SES in v1.1 if needed

**Status**: ✅ Decision made, captured in Assumptions

---

### D10: Admin UI Location — Same Next.js App (Not Separate)
**Decision**: Build admin dashboard in the same Next.js frontend app (separate route `/admin/*`), not a separate admin panel.

**Rationale**: Simpler deployment, shared codebase, faster iteration. Reduces operational complexity (one app server vs two). Route protection via JWT role validation. Separate admin panel can be refactored in v1.1 if needed.

**Implications**:
- Admin routes: `/admin/dashboard`, `/admin/products`, `/admin/orders` (protected by auth middleware)
- Auth: Same JWT token system; role/admin flag in user entity
- Deployment: Single Next.js app serves both shopper + admin
- Future: Separate admin SPA in v1.1 if organizational needs evolve

**Status**: ✅ Decision made, captured in Assumptions
